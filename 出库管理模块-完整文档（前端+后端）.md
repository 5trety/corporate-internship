## 1. 模块概述

出库管理模块是WMS系统的重要扩展功能，**独立控制出库业务**，但**共享入库功能的库存数据**（看板、库存追溯、当前库存等表）。

### 模块定位

```
入库模块（已有）          出库模块（新增）
├── inbound_order  ───┐
├── kanban         ───┤   ├── outbound_order (独立)
├── current_inventory┤   ├── outbound_detail (独立)
└── inventory_trace ──┘   └── 共享入库数据（看板、库存、追溯）
```

### 数据关系

| 功能 | 入库模块 | 出库模块 | 共享 |
|------|---------|---------|------|
| 订单管理 | inbound_order | **outbound_order** | ❌ 独立 |
| 看板管理 | 创建看板 | 更新看板状态 | ✅ 共享kanban表 |
| 库存管理 | 增加库存 | 减少库存 | ✅ 共享current_inventory |
| 追溯记录 | INBOUND记录 | OUTBOUND记录 | ✅ 共享inventory_trace |
| 零件数据 | 使用零件 | 使用零件 | ✅ 共享part表 |
| 仓库数据 | 使用仓库 | 使用仓库 | ✅ 共享warehouse表 |

## 2. 核心特性

### ✅ 出库单管理（独立控制）
- 创建出库单（支持多明细）
- 查询出库单列表（分页、筛选）
- 查看出库单详情
- 删除出库单（仅待出库状态）

### ✅ 扫码出库（共享库存数据）
- 验证看板是否可以出库
- **先进先出（FIFO）强制检查**
- 更新当前库存（`current_inventory`）
- 更新看板状态（`stored` → `outbound`）
- 记录出库追溯（`inventory_trace`）
- 关联出库单自动更新进度

### ✅ FIFO推荐系统
- 按入库时间排序看板
- 智能计算需要哪些看板
- 返回推荐出库方案

### ✅ 库存追溯
- 查询所有出库记录
- 按看板追溯完整流转（入库+出库）
- 支持多条件筛选

### ✅ 出库二维码生成
- 为出库单生成独立二维码
- 支持批量生成二维码
- 二维码内容包含：出库单号、零件号、数量、时间戳
- 前端可使用 qrcode 库生成图片并打印

## 3. 数据库设计

### 3.1 outbound_order（出库单表）

```sql
CREATE TABLE IF NOT EXISTS `outbound_order` (
  `id` int NOT NULL AUTO_INCREMENT,
  `order_no` varchar(50) NOT NULL,
  `outbound_type` varchar(50) NOT NULL COMMENT '出库类型：销售出库、退货出库、生产领料等',
  `customer_code` varchar(50) DEFAULT NULL COMMENT '客户编码',
  `customer_name` varchar(100) DEFAULT NULL COMMENT '客户名称',
  `warehouse_code` varchar(50) DEFAULT NULL COMMENT '仓库编码',
  `status` varchar(20) DEFAULT 'pending' COMMENT '状态：pending待出库, partial部分出库, completed已完成',
  `total_quantity` int DEFAULT 0 COMMENT '总数量',
  `shipped_quantity` int DEFAULT 0 COMMENT '已出库数量',
  `total_boxes` int DEFAULT 0 COMMENT '总箱数',
  `shipped_boxes` int DEFAULT 0 COMMENT '已出库箱数',
  `remark` text COMMENT '备注',
  `created_by` varchar(50) DEFAULT NULL COMMENT '创建人',
  `created_at` datetime DEFAULT CURRENT_TIMESTAMP,
  `updated_at` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_order_no` (`order_no`),
  KEY `idx_status` (`status`),
  KEY `idx_created_at` (`created_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='出库单表';
```

**字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| order_no | varchar(50) | 出库单号（唯一），格式：OUT+时间戳+随机数 |
| outbound_type | varchar(50) | 出库类型：销售出库、退货出库、生产领料等 |
| customer_code | varchar(50) | 客户编码 |
| customer_name | varchar(100) | 客户名称 |
| warehouse_code | varchar(50) | 仓库编码 |
| status | varchar(20) | 状态：pending/partial/completed |
| total_quantity | int | 总数量 |
| shipped_quantity | int | 已出库数量 |
| total_boxes | int | 总箱数 |
| shipped_boxes | int | 已出库箱数 |

### 3.2 outbound_order_detail（出库单明细表）

```sql
CREATE TABLE IF NOT EXISTS `outbound_order_detail` (
  `id` int NOT NULL AUTO_INCREMENT,
  `order_no` varchar(50) NOT NULL COMMENT '出库单号',
  `part_code` varchar(100) NOT NULL COMMENT '零件号',
  `part_name` varchar(200) DEFAULT NULL COMMENT '零件名称',
  `expected_quantity` int DEFAULT 0 COMMENT '预期出库数量',
  `shipped_quantity` int DEFAULT 0 COMMENT '已出库数量',
  `packaging_capacity` int DEFAULT 0 COMMENT '包装容量',
  `expected_boxes` int DEFAULT 0 COMMENT '预期箱数',
  `shipped_boxes` int DEFAULT 0 COMMENT '已出库箱数',
  `unit` varchar(10) DEFAULT '个' COMMENT '单位',
  PRIMARY KEY (`id`),
  KEY `idx_order_no` (`order_no`),
  KEY `idx_part_code` (`part_code`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='出库单明细表';
```

### 3.3 状态说明

#### 出库单状态
| 状态码 | 名称 | 说明 | 自动流转 |
|--------|------|------|---------|
| pending | 待出库 | 刚创建，未开始出库 | shipped_quantity = 0 |
| partial | 部分出库 | 已出库部分数量 | 0 < shipped < total |
| completed | 已完成 | 全部出库完成 | shipped >= total |

#### 看板状态（新增outbound）
| 状态码 | 名称 | 说明 | 可出库 |
|--------|------|------|--------|
| pending | 待入库 | 已创建入库单，未扫码 | ❌ |
| scanned | 已扫码 | 已扫码，未确认入库 | ❌ |
| stored | 已入库 | 已入库上架 | ✅ **可以出库** |
| **outbound** | **已出库** | **已全部出库** | ❌ |

#### 追溯记录类型
| 类型 | 名称 | 说明 |
|------|------|------|
| INBOUND | 入库 | 入库操作 |
| **OUTBOUND** | **出库** | **出库操作** |
| MOVE | 移库 | 库位移动 |

#### 6. 扫码出库（核心接口）

```
POST /api/wms/outbound/scan/outbound
```

**请求体**：
```json
{
  "kanbanNo": "KB20260610001",
  "orderNo": "OUT202606101234560001",
  "outboundQuantity": 50
}
```

**参数说明**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| kanbanNo | string | 是 | 看板号 |
| orderNo | string | 否 | 关联的出库单号 |
| outboundQuantity | int | 否 | 出库数量（不传则全部出库） |

**前端请求示例**：
```javascript
import request from '@/api/http'

const scanOutbound = async (data) => {
  return await request.post('/api/wms/outbound/scan/outbound', data)
}

// 调用示例
const res = await scanOutbound({
  kanbanNo: 'KB20260610001',
  orderNo: 'OUT202606101234560001',
  outboundQuantity: 50
})
```

**功能流程**：
1. 验证看板状态
2. **FIFO检查**：查询是否有更早入库的同零件看板
3. 更新当前库存（`current_inventory`）
4. 更新看板状态（`stored` → `outbound` 或保持`stored`）
5. 记录出库追溯（`inventory_trace`）
6. 如果关联出库单，更新出库单明细和状态

**响应**：
```json
{
  "code": 200,
  "data": {
    "success": true,
    "kanbanNo": "KB20260610001",
    "outboundQuantity": 50,
    "traceNo": "TRACE202606101234560001"
  }
}
```

- `OUT202606141007278839`：出库单号
- `A`：零件号
- `150`：出库数量
- `20260614120000`：生成时间（yyyyMMddHHmmss）

## 8. 核心功能详解

### 8.1 先进先出（FIFO）

**概念**：出库时优先使用最早入库的库存

**实现逻辑**：
1. 查询该零件的所有已入库看板
2. 按入库时间升序排列
3. 从最早的看板开始扣减库存
4. 如果最早看板数量不足，继续从下一个看板扣减

**前端应用**：
- 扫码出库页面显示 FIFO 推荐列表
- 帮助操作人员选择正确的看板

### 8.2 出库单状态流转

```
pending（待出库）
    ↓
partial（部分出库）←── 部分出库完成
    ↓
completed（已完成）←── 全部出库完成
```

**状态更新时机**：
- `pending → partial`：首次扫码出库
- `partial → completed`：全部数量出库完成

### 8.3 看板状态更新

**出库前**：
- 看板状态必须为 `stored`（已入库）
- 库存数量必须大于 0

**出库后**：
- 如果全部出库：看板状态更新为 `outbound`
- 如果部分出库：看板状态保持 `stored`，更新库存数量

## 11. 常见问题

### Q1：出库失败，提示"违反FIFO规则"？

**原因**：存在更早入库的同零件看板未出库

**解决方案**：
1. 查询该零件的 FIFO 推荐列表
2. 按推荐顺序先出库更早入库的看板
3. 然后再出库当前看板

### Q2：出库数量超过库存数量？

**原因**：输入的出库数量大于当前库存

**解决方案**：
1. 核实实际库存数量
2. 调整出库数量
3. 重新执行出库

### Q3：无法删除已完成的出库单？

**原因**：已完成或部分出库的订单不能删除

**解决方案**：
- 已出库的库存不能回退
- 如需处理异常，请联系管理员

### Q4：前端请求出现 415 错误？

**原因**：后端没有正确配置 JSON 解析

**解决方案**：
1. 确保请求头设置正确
2. 前端添加 `Content-Type: application/json`
3. 后端确保 `@RequestBody` 注解

### Q5：二维码扫描失败？

**检查项**：
1. 二维码内容格式是否正确
2. 扫描设备是否正常
3. 二维码是否清晰可读
