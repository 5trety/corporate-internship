## 1. 模块概述

出库管理模块是WMS系统的重要扩展功能，**独立控制出库业务**，但**共享入库功能的库存数据**（看板、库存追溯、当前库存等表）。

### 模块定位

```
入库模块（已有）          出库模块（新增）
├── inbound_order  ───┐
├── kanban         ───┤   ├── outbound_order (独立)
├── current_inventory┤   ├── outbound_detail (独立)
└── inventory_trace ──┘   └── 共享入库数据（看板、库存、追溯）
```

### 数据关系

| 功能 | 入库模块 | 出库模块 | 共享 |
|------|---------|---------|------|
| 订单管理 | inbound_order | **outbound_order** | ❌ 独立 |
| 看板管理 | 创建看板 | 更新看板状态 | ✅ 共享kanban表 |
| 库存管理 | 增加库存 | 减少库存 | ✅ 共享current_inventory |
| 追溯记录 | INBOUND记录 | OUTBOUND记录 | ✅ 共享inventory_trace |
| 零件数据 | 使用零件 | 使用零件 | ✅ 共享part表 |
| 仓库数据 | 使用仓库 | 使用仓库 | ✅ 共享warehouse表 |

## 2. 核心特性

### ✅ 出库单管理（独立控制）
- 创建出库单（支持多明细）
- 查询出库单列表（分页、筛选）
- 查看出库单详情
- 删除出库单（仅待出库状态）

### ✅ 扫码出库（共享库存数据）
- 验证看板是否可以出库
- **先进先出（FIFO）强制检查**
- 更新当前库存（`current_inventory`）
- 更新看板状态（`stored` → `outbound`）
- 记录出库追溯（`inventory_trace`）
- 关联出库单自动更新进度

### ✅ FIFO推荐系统
- 按入库时间排序看板
- 智能计算需要哪些看板
- 返回推荐出库方案

### ✅ 库存追溯
- 查询所有出库记录
- 按看板追溯完整流转（入库+出库）
- 支持多条件筛选

### ✅ 出库二维码生成
- 为出库单生成独立二维码
- 支持批量生成二维码
- 二维码内容包含：出库单号、零件号、数量、时间戳
- 前端可使用 qrcode 库生成图片并打印

## 3. 数据库设计

### 3.1 outbound_order（出库单表）

```sql
CREATE TABLE IF NOT EXISTS `outbound_order` (
  `id` int NOT NULL AUTO_INCREMENT,
  `order_no` varchar(50) NOT NULL,
  `outbound_type` varchar(50) NOT NULL COMMENT '出库类型：销售出库、退货出库、生产领料等',
  `customer_code` varchar(50) DEFAULT NULL COMMENT '客户编码',
  `customer_name` varchar(100) DEFAULT NULL COMMENT '客户名称',
  `warehouse_code` varchar(50) DEFAULT NULL COMMENT '仓库编码',
  `status` varchar(20) DEFAULT 'pending' COMMENT '状态：pending待出库, partial部分出库, completed已完成',
  `total_quantity` int DEFAULT 0 COMMENT '总数量',
  `shipped_quantity` int DEFAULT 0 COMMENT '已出库数量',
  `total_boxes` int DEFAULT 0 COMMENT '总箱数',
  `shipped_boxes` int DEFAULT 0 COMMENT '已出库箱数',
  `remark` text COMMENT '备注',
  `created_by` varchar(50) DEFAULT NULL COMMENT '创建人',
  `created_at` datetime DEFAULT CURRENT_TIMESTAMP,
  `updated_at` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_order_no` (`order_no`),
  KEY `idx_status` (`status`),
  KEY `idx_created_at` (`created_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='出库单表';
```

**字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| order_no | varchar(50) | 出库单号（唯一），格式：OUT+时间戳+随机数 |
| outbound_type | varchar(50) | 出库类型：销售出库、退货出库、生产领料等 |
| customer_code | varchar(50) | 客户编码 |
| customer_name | varchar(100) | 客户名称 |
| warehouse_code | varchar(50) | 仓库编码 |
| status | varchar(20) | 状态：pending/partial/completed |
| total_quantity | int | 总数量 |
| shipped_quantity | int | 已出库数量 |
| total_boxes | int | 总箱数 |
| shipped_boxes | int | 已出库箱数 |

### 3.2 outbound_order_detail（出库单明细表）

```sql
CREATE TABLE IF NOT EXISTS `outbound_order_detail` (
  `id` int NOT NULL AUTO_INCREMENT,
  `order_no` varchar(50) NOT NULL COMMENT '出库单号',
  `part_code` varchar(100) NOT NULL COMMENT '零件号',
  `part_name` varchar(200) DEFAULT NULL COMMENT '零件名称',
  `expected_quantity` int DEFAULT 0 COMMENT '预期出库数量',
  `shipped_quantity` int DEFAULT 0 COMMENT '已出库数量',
  `packaging_capacity` int DEFAULT 0 COMMENT '包装容量',
  `expected_boxes` int DEFAULT 0 COMMENT '预期箱数',
  `shipped_boxes` int DEFAULT 0 COMMENT '已出库箱数',
  `unit` varchar(10) DEFAULT '个' COMMENT '单位',
  PRIMARY KEY (`id`),
  KEY `idx_order_no` (`order_no`),
  KEY `idx_part_code` (`part_code`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='出库单明细表';
```

### 3.3 状态说明

#### 出库单状态
| 状态码 | 名称 | 说明 | 自动流转 |
|--------|------|------|---------|
| pending | 待出库 | 刚创建，未开始出库 | shipped_quantity = 0 |
| partial | 部分出库 | 已出库部分数量 | 0 < shipped < total |
| completed | 已完成 | 全部出库完成 | shipped >= total |

#### 看板状态（新增outbound）
| 状态码 | 名称 | 说明 | 可出库 |
|--------|------|------|--------|
| pending | 待入库 | 已创建入库单，未扫码 | ❌ |
| scanned | 已扫码 | 已扫码，未确认入库 | ❌ |
| stored | 已入库 | 已入库上架 | ✅ **可以出库** |
| **outbound** | **已出库** | **已全部出库** | ❌ |

#### 追溯记录类型
| 类型 | 名称 | 说明 |
|------|------|------|
| INBOUND | 入库 | 入库操作 |
| **OUTBOUND** | **出库** | **出库操作** |
| MOVE | 移库 | 库位移动 |

### 二、扫码出库

#### 5. 验证看板是否可以出库

```
GET /wms/outbound/kanban/validate/{kanbanNo}
```

**功能**：
- 检查看板是否存在
- 检查看板状态是否为`stored`（已入库）
- 检查库存数量是否大于0
- 返回入库时间（用于FIFO判断）

**响应**：
```json
{
  "code": 200,
  "data": {
    "kanban_no": "KB20260610001",
    "part_code": "PART001",
    "part_name": "零件A",
    "current_quantity": 100,
    "location_code": "LOC-A01",
    "status": "stored",
    "inbound_time": "2026-06-10 10:00:00"
  }
}
```

---

#### 6. 扫码出库（核心接口）

```
POST /wms/outbound/scan/outbound
```

**请求体**：
```json
{
  "kanbanNo": "KB20260610001",
  "orderNo": "OUT202606101234560001",
  "outboundQuantity": 50
}
```

**参数说明**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| kanbanNo | string | 是 | 看板号 |
| orderNo | string | 否 | 关联的出库单号 |
| outboundQuantity | int | 否 | 出库数量（不传则全部出库） |

**功能流程**：
1. 验证看板状态
2. **FIFO检查**：查询是否有更早入库的同零件看板
3. 更新当前库存（`current_inventory`）
4. 更新看板状态（`stored` → `outbound` 或保持`stored`）
5. 记录出库追溯（`inventory_trace`）
6. 如果关联出库单，更新出库单明细和状态

**响应**：
```json
{
  "code": 200,
  "data": {
    "success": true,
    "kanbanNo": "KB20260610001",
    "outboundQuantity": 50,
    "traceNo": "TRACE202606101234560001"
  }
}
```

### 四、出库二维码生成

#### 10. 打印出库单（含二维码数据）

```
GET /wms/outbound/print/outbound-order/{orderNo}
```

**功能**：
- 获取出库单详情
- 为每个明细生成二维码内容
- 返回可打印的数据

**响应**：
```json
{
  "code": 200,
  "data": {
    "order": {
      "order_no": "OUT202606141007278839",
      "outbound_type": "销售出库",
      "customer_name": "测试客户",
      "warehouse_name": "A仓库",
      "status": "pending",
      "total_quantity": 150
    },
    "details": [
      {
        "part_code": "A",
        "part_name": "abc",
        "expected_quantity": 150,
        "packaging_capacity": 10,
        "expected_boxes": 15,
        "unit": "个",
        "qrContent": "OUTBOUND|OUT202606141007278839|A|150|20260614120000",
        "qrType": "OUTBOUND"
      }
    ],
    "qrType": "OUTBOUND_ORDER"
  }
}
```

**二维码内容格式**：
```
OUTBOUND|出库单号|零件号|数量|时间戳
```

**示例**：
```
OUTBOUND|OUT202606141007278839|A|150|20260614120000
```

**字段说明**：
- `OUTBOUND`：标识这是出库二维码
- `OUT202606141007278839`：出库单号
- `A`：零件号
- `150`：出库数量
- `20260614120000`：生成时间（yyyyMMddHHmmss）

## 7. 核心功能详解

### 7.1 先进先出（FIFO）实现

#### 核心逻辑

```java
// 1. 查询当前看板的入库时间
SELECT action_time FROM inventory_trace
WHERE kanban_no = ? AND action_type = 'INBOUND'
ORDER BY action_time ASC LIMIT 1

// 2. 检查是否有更早入库的同零件看板
SELECT COUNT(*) FROM kanban k
LEFT JOIN current_inventory c ON k.kanban_no = c.kanban_no
WHERE k.part_code = ?           -- 同零件
  AND k.status = 'stored'       -- 已入库
  AND c.quantity > 0            -- 有库存
  AND k.created_at < ?          -- 比当前看板早
  AND k.kanban_no != ?          -- 不是当前看板

// 3. 如果count > 0，返回错误提示
"存在更早入库的同零件看板，请优先出库（先进先出原则）"
```

#### FIFO推荐算法

```java
// 输入：零件编码 PART001，需要数量 150
// 输出：按入库时间排序的看板列表

SELECT k.*, c.quantity, inbound_time
FROM kanban k
LEFT JOIN current_inventory c ON k.kanban_no = c.kanban_no
WHERE k.part_code = 'PART001'
  AND k.status = 'stored'
  AND c.quantity > 0
ORDER BY inbound_time ASC  // 入库时间升序

// 计算每个看板的使用数量
remaining = 150
for kanban in kanbans:
    useQty = min(remaining, kanban.quantity)
    kanban.useQuantity = useQty
    remaining -= useQty
    if remaining <= 0: break
```

### 7.2 库存更新流程

```sql
-- 1. 减少当前库存
UPDATE current_inventory 
SET quantity = quantity - ?, updated_at = NOW()
WHERE kanban_no = ?

-- 2. 更新看板状态
UPDATE kanban 
SET status = CASE 
    WHEN quantity - ? <= 0 THEN 'outbound' 
    ELSE 'stored' 
  END,
  scan_time = NOW(),
  scan_by = ?
WHERE kanban_no = ?

-- 3. 记录追溯
INSERT INTO inventory_trace (
    trace_no, kanban_no, order_no, part_code, 
    part_name, supplier_code, quantity, location_code, 
    action_type, operator, remark
) VALUES (?, ?, ?, ?, ?, ?, ?, ?, 'OUTBOUND', ?, ?)

-- 4. 更新出库单明细
UPDATE outbound_order_detail 
SET shipped_quantity = shipped_quantity + ?,
    shipped_boxes = shipped_boxes + CEIL(? / NULLIF(packaging_capacity, 0))
WHERE order_no = ? AND part_code = ?

-- 5. 更新出库单状态
UPDATE outbound_order o
SET shipped_quantity = (
    SELECT COALESCE(SUM(shipped_quantity), 0) 
    FROM outbound_order_detail WHERE order_no = ?
  ),
  shipped_boxes = (
    SELECT COALESCE(SUM(shipped_boxes), 0) 
    FROM outbound_order_detail WHERE order_no = ?
  ),
  status = CASE 
    WHEN shipped_quantity >= total_quantity THEN 'completed'
    WHEN shipped_quantity > 0 THEN 'partial'
    ELSE 'pending' 
  END,
  updated_at = NOW()
WHERE order_no = ?
```

---

### 7.3 数据流转图

```
创建出库单
  └─> outbound_order (插入主单)
  └─> outbound_order_detail (插入明细)

扫码出库
  ├─> 验证看板 (kanban表)
  ├─> FIFO检查 (查询更早入库的看板)
  ├─> 更新库存 (current_inventory.quantity减少)
  ├─> 更新看板状态 (kanban.status = 'outbound' 或 'stored')
  ├─> 记录追溯 (inventory_trace插入OUTBOUND记录)
  └─> 更新出库单 (outbound_order_detail和outbound_order)

出库单状态自动更新
  ├─> shipped_quantity = 0          → pending (待出库)
  ├─> 0 < shipped_quantity < total  → partial (部分出库)
  └─> shipped_quantity >= total     → completed (已完成)
```

## 9. 注意事项

### ⚠️ 重要提示

1. **FIFO强制执行**：系统会严格检查先进先出，无法跳过
2. **库存实时同步**：出库后立即更新，保证数据准确性
3. **追溯完整记录**：每次出库都有完整记录，支持审计
4. **出库单可选**：扫码出库可以不关联出库单（直接出库）
5. **状态自动流转**：出库单状态根据出库进度自动更新
6. **需要登录**：所有接口都需要先登录（Session验证）

### 🔒 权限控制

当前版本使用Session验证登录状态：

## 10. 常见问题

### Q1: 为什么出库时提示"存在更早入库的同零件看板"？

**A**: 这是FIFO（先进先出）机制在起作用。系统检测到有入库时间更早的同零件看板还有库存，为了保证库存周转合理，强制要求优先出库早入库的看板。

**解决方案**：
- 先查询FIFO推荐列表：`GET /wms/outbound/kanban/fifo-list?partCode=xxx`
- 按照推荐顺序依次出库

---

### Q2: 出库单状态为什么没有自动更新？

**A**: 出库单状态是根据`shipped_quantity`（已出库数量）自动计算的：
- `shipped_quantity = 0` → `pending`
- `0 < shipped_quantity < total_quantity` → `partial`
- `shipped_quantity >= total_quantity` → `completed`

**检查方法**：
```sql
SELECT order_no, total_quantity, shipped_quantity, status 
FROM outbound_order 
WHERE order_no = 'OUT001';
```

---

### Q3: 看板出库后状态是什么？

**A**: 根据出库数量决定：
- 全部出库（出库数量 = 库存数量）→ `outbound`
- 部分出库（出库数量 < 库存数量）→ 保持`stored`

---

### Q4: 如何追溯某个看板的完整流转历史？

**A**: 使用追溯接口：
```bash
GET /wms/outbound/trace/by-kanban/{kanbanNo}
```

这会返回该看板的所有操作记录（入库、出库、移库等），按时间顺序排列。

---

### Q5: 出库单删除失败怎么办？

**A**: 只有`pending`（待出库）状态的出库单才能删除。如果已经开始出库（状态为`partial`或`completed`），则不允许删除。

**解决方案**：
- 检查出库单状态：`GET /wms/outbound/detail/{orderNo}`
- 只有`status = "pending"`时才能删除

---

### Q6: 如何批量出库？

**A**: 当前版本支持逐个看板扫码出库。如需批量出库，可以：
1. 循环调用扫码出库接口
2. 或者扩展后端添加批量出库接口

### 11.2 核心业务逻辑

#### 1. 创建出库单流程
```
接收JSON请求
  ↓
参数类型安全检查（details数组解析）
  ↓
生成出库单号（OUT+时间戳+随机数）
  ↓
获取当前登录用户（Session）
  ↓
计算总数量和总箱数
  ↓
插入出库单主表（status='pending'）
  ↓
插入出库单明细（shipped_quantity=0）
  ↓
返回orderNo
```

#### 2. 扫码出库流程（核心）
```
接收出库请求（kanbanNo, orderNo, outboundQuantity）
  ↓
1. 验证看板
   - 看板是否存在
   - 状态是否为stored
   - 库存数量是否>0
   - 获取入库时间（用于FIFO）
  ↓
2. 检查出库数量
   - 未传数量 → 默认全部出库
   - 超出库存 → 返回错误
  ↓
3. FIFO先进先出检查
   - 查询同零件、更早入库、有库存的看板
   - 存在 → 拒绝出库
   - 不存在 → 继续
  ↓
4. 更新库存（current_inventory.quantity减少）
  ↓
5. 更新看板状态
   - 全部出库 → status='outbound'
   - 部分出库 → status='stored'
  ↓
6. 记录追溯（inventory_trace插入OUTBOUND记录）
  ↓
7. 如果关联出库单，更新出库单进度
   - 更新明细：shipped_quantity, shipped_boxes
   - 更新主单：重新计算总出库量，自动更新status
  ↓
返回成功（kanbanNo, outboundQuantity, traceNo）
```

#### 3. FIFO先进先出机制
```sql
-- 查询是否有更早入库的同零件看板

LEFT JOIN current_inventory c ON k.kanban_no = c.kanban_no
WHERE k.part_code = ?           -- 同零件
  AND k.status = 'stored'       -- 已入库
  AND c.quantity > 0            -- 有库存
  AND k.created_at < ?          -- 比当前看板早
  AND k.kanban_no != ?          -- 不是当前看板

-- 如果count > 0，拒绝出库
```

**设计亮点**：
- ✅ 强制FIFO，无法跳过
- ✅ 基于入库时间（inventory_trace.action_time）
- ✅ 只检查有库存的看板
- ✅ 提供FIFO推荐接口辅助用户决策
