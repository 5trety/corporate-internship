# WMS出库管理系统 - 完整说明文档

## 一、系统概述

WMS出库管理系统是仓储管理系统的重要组成部分,与入库系统共享底层数据,实现完整的仓储管理闭环。系统支持出库单管理、扫码出库、FIFO先进先出、库存联动更新、追溯记录等核心功能。

### 1.1 系统特点

- **独立模块**: 出库功能与入库功能分离控制
- **数据共享**: 共享库存、零件、供应商、看板等基础数据
- **FIFO原则**: 严格执行先进先出,优先出库最早入库的库存
- **库存联动**: 出库自动扣减库存,更新看板状态
- **完整追溯**: 记录每次出库操作,支持历史查询
- **响应式设计**: 支持PC端和移动端自适应显示
- **摄像头扫码**: 支持手机/平板摄像头扫码出库

## 二、数据库设计

### 2.1 出库单表 (outbound_order)

```sql
CREATE TABLE IF NOT EXISTS `outbound_order` (
  `id` int NOT NULL AUTO_INCREMENT,
  `order_no` varchar(50) NOT NULL,
  `outbound_type` varchar(50) NOT NULL COMMENT '出库类型:销售出库、退货出库、生产领料等',
  `customer_code` varchar(50) DEFAULT NULL COMMENT '客户编码',
  `customer_name` varchar(100) DEFAULT NULL COMMENT '客户名称',
  `warehouse_code` varchar(50) DEFAULT NULL COMMENT '仓库编码',
  `status` varchar(20) DEFAULT 'pending' COMMENT '状态:pending待出库, partial部分出库, completed已完成',
  `total_quantity` int DEFAULT 0 COMMENT '总数量',
  `shipped_quantity` int DEFAULT 0 COMMENT '已出库数量',
  `total_boxes` int DEFAULT 0 COMMENT '总箱数',
  `shipped_boxes` int DEFAULT 0 COMMENT '已出库箱数',
  `remark` text COMMENT '备注',
  `created_by` varchar(50) DEFAULT NULL COMMENT '创建人',
  `created_at` datetime DEFAULT CURRENT_TIMESTAMP,
  `updated_at` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_order_no` (`order_no`),
  KEY `idx_status` (`status`),
  KEY `idx_created_at` (`created_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='出库单表';
```

**字段说明:**
- `order_no`: 出库单号,格式: SO-yyyyMMdd-0001
- `outbound_type`: 出库类型(销售出库、退货出库、生产领料、调拨出库)
- `status`: 出库单状态(pending/partial/completed)
- `total_quantity`: 计划出库总数量
- `shipped_quantity`: 实际已出库数量
- `total_boxes`: 计划总箱数
- `shipped_boxes`: 实际已出库箱数

CREATE TABLE IF NOT EXISTS `outbound_order_detail` (

#### 3.2.2 扫码出库

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 扫码出库 | POST | `/wms/scan/outbound` | 执行FIFO出库操作 |

**接口示例:**
```json
POST /wms/scan/outbound
{
  "orderNo": "SO-20260614-0001",
  "partCode": "P001",
  "quantity": 50
}
```

**核心逻辑:**
1. 验证出库单状态(不能是completed)
2. 查询库存(`current_inventory`),按入库时间ASC排序(FIFO)
3. 校验库存数量是否充足
4. 按FIFO原则扣减库存
5. 更新看板状态为`outbound`
6. 记录追溯信息到`inventory_trace`
7. 更新出库单的`shipped_quantity`
8. 判断是否完成,更新出库单状态

**返回:**
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "success": true,
    "message": "出库成功",
    "orderNo": "SO-20260614-0001",
    "partCode": "P001",
    "quantity": 50,
    "boxes": 1
  }
}
```

## 五、核心业务流程

### 5.1 出库单创建流程

```
1. 用户进入"创建出库单"页面
2. 填写出库单信息:
   - 出库类型(销售/退货/生产领料/调拨)
   - 客户编码、客户名称
   - 目标仓库
   - 备注
3. 添加出库明细:
   - 搜索选择零件
   - 输入预期出库数量
   - 系统自动计算箱数
4. 提交保存
5. 系统生成出库单号: SO-yyyyMMdd-0001
6. 状态设为"pending"(待出库)
```

### 5.2 打印看板流程

```
1. 在出库单列表点击"打印看板"
2. 系统加载出库明细
3. 为每个零件生成:
   - 唯一看板号: KAN-{timestamp}-{partCode}
   - 二维码(内容只看板号)
4. 保存到kanban表,状态为"pending"
5. 弹出打印窗口
6. 用户打印看板贴纸
```

### 5.3 扫码出库流程 (核心)

```
1. 用户进入"扫码出库"页面
2. 选择待出库的出库单
3. 扫描看板二维码(或手动输入看板号)
4. 系统验证:
   ✓ 看板是否存在
   ✓ 看板状态是否为"pending"或"stored"
   ✓ 出库单状态是否为"pending"或"partial"
5. 显示看板信息:
   - 零件号、零件名称
   - 供应商
   - 库存数量
   - 入库时间
6. 输入本次出库数量
7. 点击"确认出库(FIFO)"
8. 系统执行:
   a. 查询该零件的所有库存(按入库时间ASC)
   b. 计算总可用库存
   c. 校验: 可用库存 >= 出库数量
   d. 按FIFO原则依次扣减库存:
      - 从最早入库的库存开始扣
      - 扣完一个再扣下一个
      - 库存为0时删除记录
   e. 更新看板状态为"outbound"
   f. 记录追溯信息到inventory_trace
   g. 更新出库单的shipped_quantity
   h. 判断是否完成:
      - shipped_quantity == total_quantity → status="completed"
      - 否则 → status="partial"
9. 返回成功信息
```

### 5.4 FIFO先进先出详解

**示例:**

库存情况:
```
入库批次1: 零件P001, 数量100, 入库时间: 2026-06-01 10:00
入库批次2: 零件P001, 数量200, 入库时间: 2026-06-05 14:00
入库批次3: 零件P001, 数量150, 入库时间: 2026-06-10 09:00
```

出库操作:
```
出库数量: 250
```

FIFO执行:
```
1. 从批次1扣减100 (剩余0, 删除记录)
2. 从批次2扣减150 (剩余50, 更新数量)
3. 批次3不动 (剩余150)

最终库存:
- 批次2: 50个 (2026-06-05入库)
- 批次3: 150个 (2026-06-10入库)
```

## 六、业务规则

### 6.1 出库前提条件

**必须同时满足:**
1. ✅ 零件已经入库(在`current_inventory`表中有记录)
2. ✅ 库存数量大于0 (`quantity > 0`)

**不满足时提示:**
- "零件 XXX 库存不足"
- "库存不足,当前可用: XX, 需要: XX"

### 6.2 状态流转

**出库单状态:**
```
pending (待出库)
   ↓ (部分出库)
partial (部分出库)
   ↓ (全部出库)
completed (已完成)
```

**看板状态:**
```
pending (待入库)
   ↓ (扫码入库)
scanned (已扫码)
   ↓ (入库完成)
stored (已上架) → 此时可出库
   ↓ (扫码出库)
outbound (已出库)
```

### 6.3 限制规则

| 操作 | 限制条件 |
|------|----------|
| 修改出库单 | 状态不能是completed |
| 删除出库单 | 状态不能是completed |
| 打印看板 | 状态不能是completed |
| 扫码出库 | 库存必须充足 |
| 扫码出库 | 出库单状态不能是completed |

### 6.4 库存联动

出库操作会同时更新:

1. **current_inventory表**
   - 扣减库存数量
   - 数量为0时删除记录

2. **kanban表**
   - 更新状态为"outbound"

3. **inventory_trace表**
   - 新增追溯记录(action_type='OUTBOUND')

4. **outbound_order表**
   - 更新shipped_quantity
   - 更新shipped_boxes
   - 更新status

5. **outbound_order_detail表**
   - 更新明细的shipped_quantity
   - 更新明细的shipped_boxes

## 八、常见问题

### 8.1 为什么提示"库存不足"?

**原因:**
- 零件还未入库
- 零件已全部出库,库存为0
- 出库数量超过可用库存

**解决:**
1. 检查"库存追溯"确认是否有入库记录
2. 检查"当前库存"确认可用数量
3. 调整出库数量或先进行入库操作

### 8.2 扫码出库扫的是什么码?

**答案:** 扫描的是**看板二维码**,不是出库单二维码。

**流程:**
1. 先创建出库单
2. 打印看板(生成看板号和二维码)
3. 将看板贴纸贴在货物上
4. 出库时扫描看板上的二维码

### 8.3 FIFO是什么意思?

**FIFO** = First In, First Out (先进先出)

**含义:** 最早入库的货物优先出库

**好处:**
- 避免货物长期积压
- 符合仓储管理最佳实践
- 确保货物周转效率

### 8.4 出库单可以修改吗?

**可以**,但有条件:
- 状态必须是`pending`或`partial`
- 状态为`completed`后不能修改

### 8.5 如何查看出库记录?

**两种方式:**
1. **出库历史页面**: 查看所有出库追溯记录
2. **库存追溯页面**: 筛选action_type='OUTBOUND'
